# 真实测试

用于验证策略、自主工具选择、验证证据等级、缺口补充和降级标注。

## 通用反馈闭环

所有验证共享同一个最小骨架，与语言、框架和具体工具无关：

```text
修改 → 观测 → 断言 → 决策
  │                          │
  └──── 失败则继续迭代 ──────┘
```

- **观测**：获取修改后的系统状态。可以是测试输出、HTTP 响应、控制台日志、浏览器 DOM 快照、截图、构建产物。
- **断言**：将观测结果与预期行为对比。优先使用项目已有的断言框架，其次用最小可重复脚本。
- **决策**：通过则继续下一步，失败则提取首个关键失败信号进入下一轮迭代。

Agent 不需要事先知道工具名。它只需要知道：我需要观测什么、已有哪些观测手段、缺什么。

## 项目信号识别

Agent 在启动验证前，必须先读取项目文件并自推导验证策略，不要假设项目类型，不要枚举工具名。

### 必须读取的信号源

| 信号源 | 推导内容 |
|--------|----------|
| `package.json` / `Cargo.toml` / `pyproject.toml` / `go.mod` / `pubspec.yaml` | 语言、框架、包管理器、现有 scripts/test 命令 |
| 现有测试目录（`__tests__`、`tests/`、`spec/`、`e2e/`） | 已有测试类型和运行方式 |
| CI 配置文件（`.github/workflows/`、`.gitlab-ci.yml`、`Jenkinsfile`） | 项目中已固化的验证命令 |
| `Makefile` / `justfile` / `scripts/` | 项目约定的任务运行器 |
| 路由/页面文件（`pages/`、`app/`、`routes/`） | 项目是否包含 Web UI |
| `Dockerfile` / `docker-compose.yml` | 是否依赖外部服务、数据库、容器 |

### 推导规则

1. **如果 scripts/test 已存在**：优先使用 `npm test`、`pnpm test`、`cargo test` 等，不要臆造命令名。
2. **如果项目有路由/页面目录但 e2e 测试目录为空或不包含浏览器测试**：存在 UI 验证缺口，Agent 应在验证结果中标注，并建议补充。
3. **如果项目有数据库/缓存/队列配置但无集成测试**：存在集成验证缺口。
4. **如果任何 scripts/test 都不存在**：Agent 应从语言标准工具推导最小验证命令。优先使用项目已有依赖或本地安装的工具（例如 `npm exec -- tsc --noEmit`、`npx --no-install tsc --noEmit`），不可用时标记为 `not_available` 并说明原因，不得跳过但也不得引入新依赖或触发网络下载。

## 验证证据等级

不枚举工具，而是定义验证的可信度等级。Agent 根据项目已有能力判断可达的最高等级，标注不可达等级及原因。每升一级，验证结论的可信度显著提高。

| 等级 | 名称 | 含义 | 典型观测方式（Agent 自判断） |
|------|------|------|---------------------------|
| L0 | 静态检查 | 代码语法/类型/结构正确 | typecheck、lint、编译器检查 |
| L1 | 单元逻辑 | 独立函数/类/状态机的行为正确 | 项目已有的单元测试框架 |
| L2 | 集成行为 | 模块间协作、数据通路、API 契约正确 | HTTP/CLI/DB 真实调用、集成测试 |
| L3 | UI 交互 | 用户可观察的界面行为正确 | 浏览器自动化、可访问性树快照、DOM 断言 |
| L4 | 视觉一致 | 渲染结果与预期一致 | 截图对比、视觉回归 |

### 等级判定规则

- Agent 必须从 L0 开始逐级向上判断并记录每一级的可达性。允许在 L0/L1 不可用时运行更高层的真实验证（例如项目无 typecheck 但可直接运行集成测试），但必须把不可达等级及原因写入 Requirement Coverage Matrix。未经判断直接假设某级可达或不可达，等同于跳过。
- 如果当前项目缺少某级验证能力，标记该级为 `not_available` 并在 Requirement Coverage Matrix 中标注原因。
- L3/L4 只适用于有 UI 的项目（Web、桌面应用、移动应用）。纯 CLI/库/后端服务项目的最高可达等级为 L2。
- 测试通过不等于验证完成。只有与当前需求直接相关的最高可达等级的验证通过，才能标记需求为 `passed`。

### 等级与交付可验证性的关系

| 最高可达等级 | 交付可验证性 | 条件 |
|-------------|-------------|------|
| L2+ 且关键需求均验证通过 | `verifiable` | 可使用成功交付模板 |
| 存在无法达到的等级但关键需求已通过可运行的最高级验证 | `partially_verifiable` | 必须已明确区分 passed/not_verified/blocked，且未验证项为非关键或已获用户接受 |
| 仅能达到 L0，或缺少任何与关键需求相关的 L1+ 行为验证 | `not_verifiable` | 必须按提前停止输出 |

## 验证策略选择

Agent 根据项目信号和验证证据等级，自推导验证策略。以下为推导路径，不是工具清单。

### 后端 / CLI / 库项目

```text
L0：运行 typecheck + lint（如果有）
L1：运行单元测试（如果有），否则运行最小编译/类型检查
L2：启动服务 → HTTP/CLI 调用 → 断言响应/输出/状态变化
     如果项目有数据库/缓存/队列 → 连接测试环境或 fixture 验证
```

### Web 前端项目

```text
L0：运行 typecheck + lint + build（确认无构建错误）
L1：运行组件/逻辑单元测试（如果有）
L3：启动 dev server → 浏览器自动化获取页面状态 → 断言 DOM/交互/控制台
     如果项目已有 e2e 测试 → 优先运行已有测试
     如果没有 → Agent 应建议补充，并用 L0+L1 先行验证
```

Agent 在选择浏览器自动化工具时，应优先：
1. 项目中已有的 e2e 测试框架
2. 如果不存在，建议用户安装，但不强制
3. 无法进行浏览器验证时，标记 L3 为 `not_available`，需求标注为 `not_verified`

### 桌面 / 移动客户端项目

```text
L0：运行 typecheck + 平台编译（确认可构建）
L1：运行逻辑单元测试（状态机、数据转换、校验规则）
L3：运行平台 UI 测试（如果有），否则标记 L3 为 not_available 并说明
```

客户端 UI 验证工具选择取决于平台，Agent 应从项目配置推导。如果无法确定工具且项目无已有 UI 测试，只验证到 L1，L3 标注为 `not_available`。

## 验证工具自主补充

Agent 发现验证缺口时，不应沉默，也不应假装能验证。应按以下规则主动建议：

### 缺口报告格式

```text
发现的验证缺口：
当前最高可达等级：L1
缺失：L3（UI 交互验证）

建议：
- 本项目为 Web 前端项目，但未配置浏览器自动化测试
- 推荐补充方式：[根据项目技术栈推荐最匹配的工具]
- 最小可验证脚本示例：[生成一个与当前需求直接相关的最小验证脚本]

是否采纳由用户决定。用户确认前，当前需求在 L3 维度标记为 not_verified。
```

### 最小验证脚本生成原则

Agent 生成最小验证脚本时必须：
- 只覆盖当前需求的一个关键行为，不预先编写完整的测试套件
- 使用项目已有语言和任务运行器，不引入新运行时
- 优先使用项目中已存在的测试文件和命名约定
- 生成后必须在项目中创建实际文件，不要只在对话中展示

## 真实反馈优先

验证必须优先追求真实反馈：

- 能运行单元测试就实际运行。
- 能运行集成测试就实际运行。
- 能启动服务就通过真实调用验证。
- 能连接测试数据库、缓存、队列就使用真实测试环境。
- 真实依赖不可用时，请求用户提供测试配置、账号、密钥、连接串或沙箱环境。

需要敏感资源时，不要猜测、伪造或写死。向用户说明：

```text
需要的资源：
为什么需要：
建议的最小权限：
可接受的沙箱替代：
无法提供时哪些验证不能完成：
```

只有用户确认后，才可以使用沙箱、容器、本地 mock 服务或 fixture。结果必须标注为"沙箱验证"，不能说成真实生产或完整集成验证。

## 测试运行纪律

真实测试不仅要求能跑起来，还要求跑完后环境可恢复、证据可复用、临时产物不污染项目。

### 服务生命周期

启动服务、dev server、数据库容器、队列、mock server、浏览器会话或后台脚本时，Agent 必须记录启动方式、监听端口、进程标识和停止方式。验证结束后必须停止本次启动的服务和后台进程。

执行规则：

- 优先使用项目前台命令或测试框架自带生命周期管理；需要后台启动时，必须保存 PID 或使用可控的 wrapper。
- 测试完成、失败提前停止、用户中断前交付时，都必须执行 cleanup。
- 如果无法确认某个进程是否由本次启动，不要盲目 kill；先用端口、命令行、工作目录和启动时间判断，不能确认时说明风险并请求用户确认。
- 浏览器自动化结束后必须关闭 page/context/browser；临时 server 必须释放端口。
- 数据库、缓存、队列等共享服务不应随意停止；只清理本次创建的 schema、fixture、topic、queue、临时数据或容器。

### 临时测试文件和产物

Agent 不得把一次性验证脚本、截图、日志、临时 fixture、下载文件或试验性测试套件长期留在仓库中。

执行规则：

- 临时文件优先写入项目约定的临时目录、`.agent-state/`、系统临时目录或测试框架产物目录。
- 如果为了验证必须创建 repo-tracked 路径下的临时文件，文件名必须带有明显临时含义，并在验证结束后删除，除非它已经被整理为有长期价值的回归测试。
- 新增测试文件必须先判断是否应合并到现有测试文件；不要每次为了一个行为重新造一套测试框架、helper 或目录结构。
- 无长期价值的测试代码必须清理；有长期价值的测试应命名为行为规格，接入项目已有测试命令，并避免重复覆盖。
- 交付前必须检查并说明临时产物处理结果：已删除、保留为回归测试、保留为证据，或因权限/风险无法清理。

### 测试流程沉淀

每次真实验证后，Agent 必须总结本次验证流程和方法，避免下一次重新摸索。

总结内容：

- 项目可用的最小验证命令，以及它覆盖的证据等级。
- 如果启动了服务：启动命令、健康检查方式、验证入口、停止方式。
- 如果使用数据库/缓存/队列/browser：连接方式、fixture 准备、断言方式、清理方式。
- 本次新增或复用的测试 seam，后续应该复用哪个文件、helper、命令或脚本。
- 不可用或被延后的验证，以及最小补齐条件。

总结去向：

- 短期任务状态写入 `state.json.validation`、Requirement Coverage Matrix 或交付总结。
- 可复用且跨任务有价值的流程，交付后通过 Skill Capture 沉淀到 `.agents/skills/` 或相关 skill 文档。
- 只对本次有效的流水账、完整日志、大段截图说明或敏感连接信息不要沉淀。

## 失败处理

验证失败时，不要先扩大修改范围。先提取最小有效信号：

- 失败命令。
- 失败测试或错误位置。
- 首个相关 stack frame。
- 实际输出和期望输出的差异。
- 最近修改中最可能相关的 diff。

基于失败信号迭代。不要通过删除断言、跳过测试、吞异常、固定返回值或只适配测试样例来制造通过。

## 交付前验证加固

所有关键需求已 `passed` 后，Agent 必须把验证目标从“证明已实现”切换为“主动寻找遗漏”。这一步不替代实现阶段验证，也不消耗实现迭代预算。

必需维度：

- `boundary`：边界值、空值、极大/极小输入、分页边界、状态边界。
- `negative`：错误输入、非法状态、权限不足、缺失资源、失败返回。
- `regression`：旧行为、兼容接口、已有脚本、历史测试和相邻功能不退化。

按项目需要追加：

- `integration`：真实 CLI / HTTP / DB / 队列 / 文件 IO 通路。
- `ui`：DOM、交互、可访问性或视觉验证。
- `concurrency`：并发提交、重复请求、幂等和竞态。
- `data`：迁移、序列化、分页、聚合、排序、空结果。
- `permission`：鉴权、角色、越权和敏感信息。

每轮验证加固必须产出以下之一：

- 新增或扩展真实行为测试，并运行通过。
- 运行一个等价的真实验证命令，记录输入、观测和断言。
- 发现缺陷并新增或重开 Requirement Coverage Matrix 中的 REQ。
- 标记该维度 `blocked / not_available`，并说明缺少的最小资源或用户决策。

只重复运行已经通过的 happy path 命令，不计入 `validation_hardening_iterations_used`。

验证加固结束后，必须补充测试流程总结：本轮实际验证了哪些维度、复用了哪些已有命令或测试 seam、清理了哪些临时资源、哪些验证方法应在后续任务中继续复用。不要为了满足加固轮次反复创建一次性脚本或重复测试同一行为。

### 成本控制

验证加固必须聪明地测，而不是机械地多跑全量测试。

- 每轮优先选择局部最小可证伪验证：单元测试、局部集成测试、单个 CLI/API 调用、最小状态机脚本或一条 UI 断言。
- 重型 e2e、浏览器全量回归、真实数据库/队列集成和完整 CI 默认不在每轮重复运行。
- 只有当前 REQ 的风险确实落在该层级、改动影响面较大、局部验证无法覆盖，或最终交付门禁需要时，才运行重型验证。
- 如果重型验证因耗时、账号、数据库、外部服务或 CI 权限延后，记录 `heavy_validation_deferred`、原因、剩余风险和用户可复现命令。
- 关键需求依赖被延后的重型验证时，交付可验证性只能是 `partially_verifiable` 或 `not_verifiable`，除非用户明确接受有限验证。

# 真实测试

用于验证策略、真实依赖、沙箱决策、敏感资源请求和未验证项标注。

## 验证顺序

优先使用项目已有命令，不要臆造脚本名。优先真实运行，不要用代码演练代替验证。

常见顺序：

```text
1. 针对性测试，例如单个 test file 或 test name
2. typecheck，例如 tsc --noEmit
3. lint
4. build
5. 全量 test
```

如果没有测试命令，运行最小有意义证明：编译、类型检查、CLI help、示例启动、静态检查或真实服务调用。必须明确说明未验证项。

## 真实反馈优先

验证必须优先追求真实反馈：

- 能跑单元测试就实际运行单元测试。
- 能跑集成测试就实际运行集成测试。
- 能启动服务就通过真实 HTTP/CLI/SDK 调用验证。
- 能连接测试数据库、缓存、队列或对象存储，就使用真实测试环境。
- 真实依赖不可用时，请用户提供测试配置、账号、密钥、连接串或沙箱环境。

需要敏感资源时，不要猜测、伪造或写死。向用户说明：

```text
需要的资源：
为什么需要：
建议的最小权限：
可接受的沙箱替代：
无法提供时哪些验证不能完成：
```

只有用户确认后，才可以使用沙箱、容器、本地 mock 服务或 fixture。结果必须标注为“沙箱验证”，不能说成真实生产或完整集成验证。

## 失败处理

验证失败时，不要先扩大修改范围。先提取最小有效信号：

- 失败命令。
- 失败测试或错误位置。
- 首个相关 stack frame。
- 实际输出和期望输出的差异。
- 最近修改中最可能相关的 diff。

基于失败信号迭代。不要通过删除断言、跳过测试、吞异常、固定返回值或只适配测试样例来制造通过。

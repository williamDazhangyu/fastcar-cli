# Zoom Out：系统视角

用于在 Context Reset Review、代码探索、PRD 验收或交付前复核时，让 Agent 从系统高度理解代码，而不是陷入局部细节。

## 定位

Agent 在代码库中工作时，容易陷入"只见树木不见森林"：盯着当前文件改来改去，却忽略了修改在整个系统中的位置和影响。

Zoom Out 强制 Agent 退后一步，用 3-5 句话描述：
- 当前代码在系统中的位置。
- 当前代码和哪些模块有交互。
- 当前修改可能影响哪些调用方或下游。

## 何时使用

### 1. Context Reset Review Gate 中

当所有关键 REQ passed 且 `fresh_eyes_required = true` 时，Agent 在执行 Context Reset Review 前必须先做 Zoom Out：

```text
1. Zoom Out：用 3-5 句话描述本次修改在系统中的位置和影响。
2. 清空对话实现细节。
3. 以"新接手项目的开发者"视角重新审视代码和 RCM。
4. 发现遗漏 → 创建新 REQ，继续迭代。无遗漏 → 继续优化或交付。
```

### 2. 探索不熟悉的代码时

当 Agent 被要求修改或理解一段不熟悉的代码时，先 Zoom Out：

```text
Agent 提问：
"这段代码在系统中的位置是什么？
- 它属于哪个模块/服务？
- 它被谁调用？
- 它依赖哪些外部模块？
- 修改它可能影响哪些下游？"
```

### 3. 实现前确认架构落点时

在 Grill Session 的架构决策步骤中，Agent 应 Zoom Out 确认代码落点：

```text
Agent 提问：
"基于系统结构，我计划把新代码放在 [位置]。
这个位置在系统中的角色是 [描述]。
它和 [模块 A]、[模块 B] 有交互。
这符合你的预期吗？"
```

### 4. 交付前复核时

交付前，Agent 应 Zoom Out 确认本次修改没有破坏系统一致性：

```text
检查清单：
- 修改是否引入了跨越模块边界的耦合？
- 修改是否改变了模块间的调用方向？
- 新代码是否重复了其他模块已有的功能？
- 修改是否在系统层面引入了不一致的命名、模式或错误处理？
```

## Zoom Out 输出格式

每次 Zoom Out 输出应包含：

```text
## Zoom Out：系统视角

### 位置
当前代码在 [模块/服务/层] 中，属于 [系统区域]。

### 交互
- 被 [调用方列表] 调用。
- 依赖 [依赖模块列表]。
- 数据流向：[描述]。

### 影响范围
修改当前代码可能影响：
- [影响点 1]
- [影响点 2]

### 一致性检查
- 命名是否与系统其他部分一致？
- 模式是否与系统约定一致？
- 是否有重复实现？
```

## 与 Context Reset Review 的关系

Zoom Out 是 Context Reset Review 的前置步骤：

| 步骤 | 目的 | 视角 |
|------|------|------|
| **Zoom Out** | 理解代码在系统中的位置和影响 | 系统视角（俯瞰） |
| **Context Reset Review** | 以新接手者视角复核实现是否符合需求 | 新人视角（从零开始） |

两者互补：Zoom Out 防止只见树木，Context Reset Review 防止思维惯性。

## 与领域语言的关系

Zoom Out 时必须使用领域语言（`Domain Glossary` 中的术语），而不是通用描述。这确保系统视角的描述精确、简洁，且与代码命名一致。

```text
BEFORE（无领域语言）：
"这个模块负责处理当用户购买课程后，把课程内容从数据库写入到文件系统的过程"

AFTER（有领域语言）：
"Materialization 模块负责课程物化：在购买完成后将课程内容写入文件系统"
```

## 降级规则

- 如果 Agent 无法读取足够多的系统文件来理解全貌，标记 `zoom_out = partial`，说明缺少哪些文件。
- 如果项目没有明确的模块边界，标记 `zoom_out = limited`，基于目录结构推断系统结构。
- 如果 Zoom Out 发现架构问题，记录到 `trace.decisions` 或 `state.decisions`，不在此步骤中修改代码。

# Grill Session：主动 Grilling 流程

用于在启动握手阶段，Agent 主动、反复追问用户，直到决策树的每个分支都清晰。替代被动等待用户提供清单的模式。

## 定位

Grill Session 是自动迭代启动握手的新标准流程。Agent 不再要求用户"请提供 AI 实现流程清单"，而是主动引导用户完成一系列决策确认。

核心原则：

- Agent 是 interviewer，用户是 domain expert。
- 每次只问一个关键决策，最多两个强相关决策。
- 每个问题必须完整描述背景、影响和默认推荐选项。
- 能从代码、文档、项目约定推断的事实不问用户。
- 用户回答后立即记录，再进入下一个问题。

## Grilling 步骤

### 步骤 1：目标对齐

**目的**：确认用户真正想要什么，而不是 Agent 猜测。

```text
Agent 提问：
"我理解你的目标是 [从用户原话推断]。请确认：
1. 这是正确的目标吗？
2. 有没有我遗漏的子目标？
3. 最终交付物是什么形式？"
```

**记录**：写入 `state.json` 的 `task.goal` 和 `task.successCriteria`。

### 步骤 2：成功标准

**目的**：把模糊目标变成可验证的 pass/fail 条件。

```text
Agent 提问：
"当任务完成时，以下哪些必须为真？
- [从目标推断的候选条件 A]
- [从目标推断的候选条件 B]
- 还有哪些我没提到的？"
```

**记录**：写入 `state.json` 的 `task.successCriteria`，每一条最终映射到 RCM 的 REQ。

### 步骤 3：非目标与边界

**目的**：明确什么不该做，防止 scope creep。

```text
Agent 提问：
"以下哪些是你明确不想做的？
- [从项目结构推断的候选非目标]
- 修改现有 API 兼容性
- 引入新依赖
- 还有其他吗？"
```

**记录**：写入 `state.json` 的 `task.nonGoals` 和 `task.scope`。

### 步骤 4：修改范围

**目的**：锁定允许修改的文件和目录。

```text
Agent 提问：
"根据当前项目结构，我判断修改范围应该是：
- [从目标推断的文件/目录列表]
这个范围对吗？有没有需要排除或加入的？"
```

**记录**：写入 `state.json` 的 `task.allowedScope`。

### 步骤 5：验证命令

**目的**：确认可用的验证手段。

```text
Agent 提问：
"我找到以下可用的验证命令：
- npm run build (exit code)
- npm test (exit code)
- npm run lint (exit code)
这些是否足够验证你的需求？还需要其他验证方式吗？"
```

**记录**：写入 `state.json` 的 `task.validationCommands`。不可用的验证标记为 `not_available`。

### 步骤 6：外部资源与依赖

**目的**：确认数据库、密钥、外部服务、网络等依赖。

```text
Agent 提问：
"这个任务是否需要以下资源？
- 数据库连接（MySQL/PostgreSQL/MongoDB/Redis）
- 外部 API 密钥或 token
- 网络访问
- 沙箱或容器环境
如果需要但当前不可用，我会标记为 blocked 并在需要时停止。"
```

**记录**：写入 `state.json` 的 `task.externalResources`，不可用的标记 `blocked`。

### 步骤 7：术语对齐

**目的**：建立共享语言，确保 Agent 和用户用同一套词汇。

```text
Agent 提问：
"我在项目中发现了以下核心概念：
- [从 AGENTS.md/README/代码中提取的术语]
请确认这些定义是否正确，有没有遗漏的关键术语？"
```

**记录**：写入 `state.json` 的 `domainGlossary`，详见 [quick-reference.md](quick-reference.md) §领域语言。

### 步骤 8：架构决策

**目的**：确认架构落点，避免实现后发现方向错了。

```text
Agent 提问：
"基于当前项目结构，我计划把新代码放在：
- [从项目结构推断的位置]
这符合你的预期吗？有没有需要遵循的架构约束？"
```

**记录**：写入 `state.json` 的 `decisions`。

### 步骤 9：迭代预算

**目的**：确认预算，最后一步。

```text
Agent 提问：
"迭代预算使用默认值：
- max_iterations = 100（普通模式最大实现轮次）
- autopilot_max_iterations = 20（Autopilot 模式最大实现轮次）
需要调整吗？"
```

**记录**：写入 `state.json` 的 `budgets`。

## 交互规则

### 提问节奏

- 每次只问一个关键决策，最多两个强相关决策。
- 不要一次性丢给用户 9 个步骤的所有问题。
- 用户回答后立即记录，然后进入下一个问题。
- 如果用户在某一步表示"你决定"，Agent 从项目约定推断并记录假设。

### 选择支设计

每个选择题应包含：

- 背景：为什么需要这个决策。
- 影响：选择不同会怎样。
- 推荐：默认推荐选项，并说明取舍。
- 选项：2-4 个互斥选择。

示例：

```text
"关于 API 兼容性，我需要你确认：
当前 `/api/users` 接口被 3 个前端页面调用。
修改它可能影响这些页面。

选项：
A. 保守兼容（推荐）：保留旧接口，新增 `/api/users/v2`
B. 直接替换：修改旧接口，同步更新前端
C. 先做只读验证：我先不改接口，只探索影响范围

你选哪个？"
```

### 跳过条件

以下情况可以跳过 Grilling，直接进入快速启动：

- 用户已在同一条消息中明确提供了目标、成功标准、范围、验证命令。
- 用户明确说"快速启动"、"直接开始"、"不用问了"。
- 任务规模为 `small` 且修改范围明确在一个文件内。

跳过时必须记录"跳过了 Grilling，假设如下：..."，并在 state 中标注。

## 与 SKILL.md 启动握手的关系

Grill Session 替代原 SKILL.md §启动握手中的"第一步：要求用户输入 AI 实现流程清单"和"第二步：询问并确认两个迭代预算"。

原启动握手：
```text
第一步：要求用户输入 AI 实现流程清单。
第二步：询问并确认两个迭代预算。
```

新启动握手（Grill Session）：
```text
Agent 按 9 步 Grill 流程主动 interview 用户：
目标 → 成功标准 → 非目标 → 范围 → 验证 → 资源 → 术语 → 架构 → 预算
```

## 与快速启动的关系

`quick` 模式中，Agent 仍应执行 Grilling，但可以更激进地从代码推断：

- 能从代码推断的步骤（如验证命令、修改范围），先给出推断再确认。
- 只有涉及产品行为、安全、兼容性或外部资源的决策才必须用户确认。
- 术语对齐可以简化：只确认 Agent 不确定的术语。

## 交付前检查

交付前确认 Grill Session 中记录的所有决策是否仍然有效：

- 成功标准是否全部满足。
- 非目标是否被意外触及。
- 修改范围是否被遵守。
- 术语是否与实际代码一致。
- 架构决策是否被遵循。

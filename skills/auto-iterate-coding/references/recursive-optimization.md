# 递归优化

用于初版实现已通过真实验证、`optimization_iterations > 0`，或用户要求 N 次递归改进时。

## 原则

递归优化只能在初版实现 GREEN 后开始。目标不是抽象完美，而是在有限预算内取得可度量提升。

优化阶段要克制：尽量少抽象，多复用已有 enum、type、interface、常量和项目约定。不要为了显得高级而引入新层级、新模式或不必要的通用化。

默认参数：

```text
optimization_iterations = 12
max_no_improvement_iterations = 3
```

## 优化循环

每轮流程：

```text
1. 记录当前 baseline：验证结果、关键 diff、复杂度和已知风险
2. 选择一个最高价值优化方向
3. 做一个最小优化修改
4. 重新运行真实验证
5. 对比 baseline 和本轮结果
6. 只有质量明确提升且风险可接受时才保留
7. 没有提升或风险变大时，停止或回退本轮修改
8. 达到 N 轮或无提升阈值时结束
```

## 质量维度

每轮只优化一个主要维度：

- 实现深度：补全关键错误处理、边界保护和初版中遗漏的已知限制。
- 正确性：行为更符合需求和边界情况。
- 测试覆盖：增加真实、有价值、面向行为的测试。
- 架构一致性：更符合用户确认的架构和代码落点。
- 模块深度：interface 更小，implementation 更深。
- 可维护性：命名、职责、重复代码和依赖方向更清晰。
- 复杂度：减少分支、状态、耦合或认知负担。
- 类型复用：复用已有 enum、type、interface 或常量，减少魔法字符串和重复类型定义。
- 性能：改善真实测量结果，而不是凭感觉优化。
- 安全性：降低敏感数据、权限、校验或注入风险。
- 可观测性：让真实故障更容易定位。
- 用户体验：改善用户可观察行为。

## 每轮报告

每轮优化结果必须可比较：

```text
优化目标：
优化前：
优化后：
运行的验证：
是否保留：
保留或停止原因：
```

不要用“更干净”“更好”当作证据，必须给出具体对比。

## 停止规则

在以下情况停止递归优化：

- 达到 `optimization_iterations`。
- 连续 `max_no_improvement_iterations` 轮没有明确提升。
- 优化收益低于修改风险。
- 下一步优化需要新的产品或架构决策。
- diff 超出约定范围。
- 无法重新运行真实验证。
- 优化需要引入新抽象但没有明确复用收益。

当优化暴露架构摩擦时，停止本轮自动优化并按 [architecture-friction.md](architecture-friction.md) 汇报。

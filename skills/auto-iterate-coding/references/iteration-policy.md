# Iteration Policy

本策略约束每轮自动迭代的目标、预算、停止和回滚。机器状态落点是 `state.json.iterationPolicy`。

## 单轮单目标

- 每轮只能处理一个明确目标、一个失败信号或一个 REQ。
- `maxGoalsPerIteration` 必须等于 `1`。
- 同一轮出现多个目标时，必须拆分、重新规划或请求用户确认。

## 变更预算

默认预算：

- `maxChangedFiles = 8`
- `maxDiffLines = 800`
- `maxNoProgressIterations = 3`

超预算处理：

- 低风险轻微超限可以记录为 Soft Rule，但不得静默继续。
- 高风险目录、跨层修改、删除/迁移重要逻辑、修改测试预期或新增依赖必须触发 `ask_user`。
- 修改范围超出 Implementation Contract 时，必须标记目标漂移并阻断或升级确认。

## 停止条件

停止边界必须按用户提供的完整目标、流程清单、Runbook、MVP 清单、成功标准和显式非目标判断。单个 Phase、子任务、最小纵切、当前失败信号清零或局部验证通过，不等于整体完成。

当任务包含多阶段清单时，必须持续维护 `完整任务清单 / 已完成任务 / 当前任务 / 剩余任务 / 整体完成状态`。只要 `剩余任务` 非空且没有触发必须停止条件，就必须继续迭代到下一项，不能按成功交付停止。

### 必须停止

以下情况必须停止：

- 完整任务清单、Runbook 或用户目标中的所有关键成功标准满足，且验证通过。
- 达到 `max_iterations` 或 `optimization_iterations`。
- 连续失败达到阈值（`max_no_progress_iterations`）。
- 验证从通过变为失败、新增 regression 未处理。
- 修改范围失控或目标漂移、diff 超过大小或范围限制。
- 验证结果无法判定。
- 产品行为、API 兼容性、迁移、权限、架构落点或外部资源需要用户决策。
- 没有可靠 feedback loop，或它不能对齐用户描述的问题。
- 优化收益低于风险。
- 看门狗触发 `stop`。
- 临时 debug instrumentation、一次性 harness、原型路由或未吸收原型仍存在，且没有用户确认的保留理由。
- 架构摩擦已确认，继续自动 patch 会扩大修改范围或破坏 public interface。

### 不能作为成功停止理由

以下情况不能作为成功停止理由：

- 当前 Phase 通过，但后续 Phase、Runbook 条目或 MVP 项仍未完成。
- 一个最小纵切通过，但用户要求的是完整模块、完整项目或"直到完成所有任务"。
- 测试、构建或 e2e 通过，但 Definition of Done 仍有 `not_verified` 或 `blocked`。
- 状态文件中的 `下一步`、`剩余任务`、`未完成 TODO` 或 `已知限制` 仍包含用户目标内的必要工作。
- 最终交付成果无法被当前环境验证，或缺少必须的验收数据、命令、权限、外部服务、数据库、密钥、沙箱替代或用户决策。
- 原型能运行或某个方案看起来可行，但尚未被用户确认、吸收为生产实现并完成验证。
- 所有 REQ passed 但 Watchdog.fresh_eyes_required = true。
- 所有 REQ passed 但 Context Reset Review Gate 尚未通过，或复核发现 Standards / Spec 问题并重开 REQ。
- 所有 REQ passed 但 validation_hardening 未达到最小轮次、缺少 boundary / negative / regression 维度，或状态仍为 pending / found_issue。

### 停止时输出

提前停止时必须输出：

```text
已完成内容：
停止原因：
当前停止阶段：
实现迭代总数：
递归优化轮次：
总循环次数：
最后一个关键错误：
已运行的验证：
未运行验证及原因：
交付可验证性：verifiable / partially_verifiable / not_verifiable / unknown
看门狗触发原因：
上下文清空复核：
未清理临时产物：
架构摩擦：
需要用户提供的资源或决策：
建议的下一步：
```

不要把停止包装成完成。完成与提前停止必须明确区分。

### 继续前检查

继续下一轮前问自己：

- 这一轮是否有新的失败信号？
- 修改范围是否仍和用户目标直接相关？
- 是否仍有可重复 feedback loop？
- 是否正在通过 mock、硬编码或跳过测试掩盖失败？
- 下一步是否需要用户决策？
- 交付成果是否真的能被当前环境验证？
- 看门狗状态是否为 clear？
- Context Reset Review Gate 是否已经只依据 state.json、原始需求、当前代码/diff、真实验证结果、项目规范和相关 skills 重新复核？
- 是否还留有 debug 日志、harness、原型或临时路由？
- 是否已经出现需要用户确认的架构摩擦？

如果任一答案显示风险上升，停止并汇报。

## 回滚策略

- 只允许回滚本轮 Agent 自己的修改。
- 不得使用破坏性 git 命令覆盖用户已有改动。
- 无法安全回滚时，保留失败证据，更新 `blockingReasons` 和 `deliveryEvidence.risks`，然后 `stop` 或 `ask_user`。
- 回滚后仍计入一次实现迭代，因为本轮已经产生修改、验证和状态更新闭环。

## 需求到实现

用于任务前半段：需求规格化、现状探索、架构确认、脚手架优先和垂直切片实现策略。

### 需求规格化

编辑代码前，先把用户需求转成可实现规格。需要明确或从现有代码中推断：用户目标；成功后的外部可观察行为；输入、输出、状态变化和错误行为；对旧 API、旧数据、旧行为和现有调用方的兼容性；明确非目标，防止范围自动扩大；不能由 Agent 猜测的产品决策。

完善契约时，不要一次性把所有不确定点丢给用户。优先采用交互式逐个确认：每次只问一个关键决策，先完整说明背景、影响和默认推荐选项，再给 2-4 个互斥选择。能从代码、测试、文档或项目约定推断的事实不要问；只询问会改变成功标准、范围、安全、兼容性、架构落点或外部资源使用的决策。

### 现状探索

生成新结构前，先阅读现有实现：入口、路由、命令、导出或 public interface；类型、schema、DTO、实体、配置、常量和环境约定；现有调用方和依赖方向；测试、fixture、示例和文档；框架封装、项目约定、错误处理和命名模式。

### 架构确认

`medium` / `large` 任务或代码落点不明确时，实现前让用户确认架构理解。确认：当前架构、框架、分层和 module seam；新需求应该落在哪个 module、目录、interface 或扩展点；是否必须使用脚手架、模板、代码生成器或项目命令。

### 通用实现策略

实现或修复前，先判断当前问题是孤立 case 还是同类问题的一个表现。默认优先修复根因、公共流程、抽象边界或可复用扩展点，不得为了某个特定失败样例写强耦合、硬编码或一次性特定解。

新增逻辑应优先通过配置、接口、策略、类型约束、公共工具函数或现有框架扩展点表达通用能力；只有明确证明影响范围极小、无复用价值且不会掩盖根因时，才允许局部特判，并在实现记录中说明理由。验证必须覆盖通用行为和边界条件，不能只覆盖当前失败样例或写“刚好通过”的测试。

测试设计、失败判断和修复验收必须基于真实运行结果、日志、断言、diff 或可复现步骤等证据。不得凭感觉推测问题已覆盖、失败原因已定位或修复已经生效；缺少证据时必须继续补充验证，或标记为 `not_verified`。

### 脚手架优先

初始化项目或新增 module/page/controller/service/entity/test 时，先找现有脚手架（`package.json` scripts、`bin/`、`templates/`、框架初始化流程），再考虑手动创建文件。只有确认没有合适脚手架后才手动创建，且必须复用现有目录结构、命名、导出、测试布局和配置约定。

### 垂直切片策略

把工作拆成垂直切片。每个切片必须包含：一个用户或调用方可观察行为；一个最小实现修改；一个验证方式；一个完成条件。优先使用 tracer bullet：先证明最小端到端路径，再补边界情况。

## Caveman Mode：超压缩通信

> 完整规范见 [contracts/output-discipline-contract.md](../contracts/output-discipline-contract.md) §3.2 caveman。

用于 Autopilot 长任务或上下文紧张时，将 Agent 输出压缩到最小，同时保持技术准确性。

### 输出格式

Caveman Mode 下，Agent 每轮输出只包含：

```text
[轮次] [动作] [结果] [下一步]
```

示例：`[3] fix login.ts token check | build=0 test=3/4 FAIL: expired token 401 | next: fix Date.now() → param now`

### 字段定义

| 字段 | 含义 | 必填 |
|------|------|------|
| `[轮次]` | 当前实现迭代编号 | 是 |
| `[动作]` | 本轮的修改摘要（< 8 词） | 是 |
| `[结果]` | 验证结果：exit code、通过/失败数、关键错误 | 是 |
| `[下一步]` | 下一轮的最小动作 | 是 |
| `[阻塞]` | 需要用户决策或资源 | 仅在 blocked 时 |
| `[预算]` | 剩余实现迭代轮次 | 每 5 轮或预算 < 5 时 |

### 停用条件

以下情况自动退出 Caveman Mode：need_decision / blocked（输出完整问题描述）；任务完成（输出完整交付总结）；提前停止；用户要求详细输出。

### 与上下文压缩的关系

Caveman Mode 是通信风格的改变，上下文压缩是状态存储的改变，领域语言减少描述 token。三者叠加在长任务中可节省 80%+ token 消耗。

### 质量要求

压缩的是输出长度，不是技术准确性。错误消息必须保留关键信息，不能因压缩而省略验证结果或跳过 Watchdog 检查。交付前必须退出 Caveman Mode，输出完整交付总结。

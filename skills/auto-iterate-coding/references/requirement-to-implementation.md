# 需求到实现

用于任务前半段：需求规格化、现状探索、架构确认、脚手架优先和垂直切片实现策略。

## 需求规格化

编辑代码前，先把用户需求转成可实现规格。

需要明确或从现有代码中推断：

- 用户目标。
- 成功后的外部可观察行为。
- 输入、输出、状态变化和错误行为。
- 对旧 API、旧数据、旧行为和现有调用方的兼容性。
- 明确非目标，防止范围自动扩大。
- 不能由 Agent 猜测的产品决策。

如果用户只说“实现某功能”，先从代码、测试、文档和现有用法推断；关键行为仍不明确时，停止并询问用户。

完善契约时，不要一次性把所有不确定点丢给用户。优先采用交互式逐个确认：每次只问一个关键决策，先完整说明背景、影响和默认推荐选项，再给 2-4 个互斥选择。能从代码、测试、文档或项目约定推断的事实不要问；只询问会改变成功标准、范围、安全、兼容性、架构落点或外部资源使用的决策。用户回答后，立即把结论写入 Decisions 或对话内状态，再继续下一个问题或进入实现。

推荐提问格式：

```text
需要确认：<一个具体决策>
背景：<已从代码/文档观察到的事实>
影响：<不同选择会影响的范围、兼容性或风险>
推荐：<推荐选项及原因>
选项：
A. <推荐选项> - <取舍>
B. <备选选项> - <取舍>
C. <保守/延后选项> - <取舍>
```

## 现状探索

生成新结构前，先阅读现有实现。

检查：

- 入口、路由、命令、导出或 public interface。
- 类型、schema、DTO、实体、配置、常量和环境约定。
- 现有调用方和依赖方向。
- 测试、fixture、示例和文档。
- 框架封装、项目约定、错误处理和命名模式。

输出简短 mental model：

```text
当前能力：
缺口：
可能修改点：
相关测试或验证方式：
```

## 架构确认

`medium` / `large` 任务或代码落点不明确时，实现在前要让用户确认架构理解。

确认：

- 当前架构、框架、分层和 module seam。
- 新需求应该落在哪个 module、目录、interface 或扩展点。
- 是否必须使用脚手架、模板、代码生成器或项目命令。
- 哪些捷径会破坏现有架构。

如果用户确认的架构与代码观察不一致，说明差异并等待决策，不要直接大范围实现。

## 脚手架优先

初始化项目或新增 module、页面、controller、service、entity、test 时，先找现有脚手架，再考虑手动创建文件。

检查：

- `package.json`、`Makefile`、`justfile`、`scripts/`、`bin/`。
- CLI help 中的 `--help`、`generate`、`new`、`create`、`scaffold`。
- 模板目录，例如 `templates/`、`examples/`、`assets/`、`boilerplate/`。
- 框架官方或项目内部约定的初始化流程。
- 同类相邻 module 的结构。

只有确认没有合适脚手架、模板或生成器后，才手动创建文件。手动创建必须复用现有目录结构、命名、导出、测试布局和配置约定。

## 垂直切片策略

把工作拆成垂直切片。每个切片必须包含：

- 一个用户或调用方可观察行为。
- 一个最小实现修改。
- 一个验证方式。
- 一个完成条件。

优先使用 tracer bullet：先证明最小端到端路径，再补边界情况。不要一开始就做大抽象、大重构或一次性实现所有分支。

## Prompt 合约

自动迭代时使用此合约：

```text
你正在通过有边界的迭代循环实现或修复代码。先把需求转成可实现规格，明确成功标准和非目标。写代码前先阅读现有实现、interface、调用方、测试和项目约定。完善契约时不要一次性丢给用户一大堆问题；只在关键行为无法推断时交互式逐个确认，每次完整描述背景和影响，并用选择题给出推荐选项。大范围实现前，让用户确认架构理解和代码落点。初始化或新增结构时，先找脚手架、模板或生成器，找不到再手动创建。验证时尽可能运行真实测试，不要把代码演练说成验证。如果真实测试需要数据库、密钥或外部服务，说明具体需要什么，或请求用户确认沙箱。每个阶段和每轮迭代后，报告当前步骤、循环计数、优化方向和结果。长任务和最终交付时，提供 Mermaid 进度图。只修改与任务直接相关的文件。每轮优先做最小可验证修改。不要假设不明确的产品行为。不要绕过架构和脚手架约定。不要删除、跳过或削弱测试来让验证通过。不要用 mock、硬编码或吞异常隐藏真实失败。不要执行破坏性 git 命令。不要无关全仓格式化或重构。连续多轮无进展时，停止并说明阻塞点。
```

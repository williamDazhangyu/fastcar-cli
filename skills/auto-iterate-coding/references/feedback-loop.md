## 反馈循环 / Feedback Loop

### 优先级

迭代前先构造反馈闭环。

优先级：

```text
1. 最小失败测试
2. 单个 CLI 命令或脚本
3. HTTP/curl 复现
4. 可重复 fixture 输入
5. 捕获 trace / payload / 事件日志后的隔离重放
6. 临时 harness，启动系统最小子集
7. 无头浏览器脚本，断言 DOM、console 或 network
8. 非确定性问题的多次循环复现
9. 差分循环，用同一输入比较旧/新版本或不同配置
10. 人工参与循环，必须结构化记录输入、操作和输出
```

feedback loop 应满足：

- 快速：优先秒级或局部验证，不要每轮默认跑完整慢套件。
- 确定：相同输入应产生相同结果；flaky 问题要提高复现率。
- 锐利：断言具体症状，不要只检查"没有崩溃"。
- 自动化：避免人工观察，除非用户明确提供人工验证结果。

如果无法建立可信 feedback loop，停止并说明尝试过什么、还需要什么 artifact 或环境。

无法建立循环时，不要继续猜修复。向用户请求最小缺口：

- 可复现环境、命令或账号权限。
- HAR、日志 dump、core dump、trace、fixture、录屏或真实 payload。
- 允许加入临时 instrumentation 的确认。
- 可接受的沙箱替代，例如本地数据库、mock server 或只读外部服务。

### 每轮输入

每轮迭代前收集：

- 原始用户目标和成功标准。
- 当前阶段编号和名称。
- 当前循环类型：实现迭代或递归优化。
- 当前计数：`implementation_iterations`、`optimization_iterations_used`、`total_cycles`。
- 当前迭代编号和剩余预算。
- 上一轮验证命令和失败摘要。
- 上一轮优化方向以及是否保留。
- 当前 Mermaid 或纯文本进度图。
- 当前 `git diff` 摘要和允许修改范围。
- 项目已有脚本，例如 `test`、`build`、`lint`、`typecheck`。

保持上下文紧凑。优先提供失败测试名、关键 stack frame、编译错误位置、相关 diff 和最近修改文件，不要反复粘贴超长日志。

### 复现与问题对齐

修 bug 时，确认 feedback loop 复现的是用户描述的问题，而不是附近的其他失败。

确认：

- 循环呈现相同症状。
- 失败有可比对证据：错误消息、输出、状态码、指标或 UI 状态。
- 修复后重新运行原始循环，而不只是跑更小测试。

如果只能复现相似问题，停止并解释差异。

对于非确定性 bug，目标是提高复现率，而不是立即得到 100% 稳定复现。可以固定随机种子、并行循环、加压、注入延迟、缩小 timing window 或重复运行 N 次。复现率低到无法调试时，停止并请求更强 artifact。

### Diagnose 六步循环

当诊断 bug、性能回归或 flaky 测试时，使用以下六步循环：

```text
1. REPRODUCE（复现）
    目的：建立可信的 pass/fail 信号。
    输入：用户报告的 bug 描述、错误日志、复现步骤。
    输出：可重复运行的复现命令或脚本，以及明确的失败症状。
    常见陷阱：复现了相似问题但不是同一个问题，导致修复方向错误。

2. MINIMISE（最小化）
    目的：缩小问题范围，排除无关因素。
    输入：复现步骤。
    输出：最小复现用例（最少的代码、数据、步骤）。
    常见陷阱：过度简化导致问题消失，或简化后问题表现不同。

3. HYPOTHESISE（假设）
    目的：列出 3-5 个可证伪的假设，按可能性排序。
    输入：最小复现用例、代码相关区域。
    输出：排序假设列表，每个假设包含可证伪预测。
    常见陷阱：假设太模糊无法证伪，或只列一个假设导致钻牛角尖。

4. INSTRUMENT（探针）
    目的：用最小代价验证或证伪当前假设。
    输入：当前假设和可证伪预测。
    输出：探针结果（debugger 断点、REPL inspection、query plan、profiler 或临时日志）。
    常见陷阱：到处撒日志而不是用精准工具；临时日志没有唯一前缀。

5. FIX（修复）
    目的：基于验证结果做最小修复。
    输入：探针结果确认的根因。
    输出：最小代码修改。
    常见陷阱：修复了症状而不是根因；修复引入了新问题。

6. REGRESSION-TEST（回归验证）
    目的：确认修复有效且不引入回归。
    输入：原始复现步骤 + 修复后的代码。
    输出：原始复现通过 + 回归测试通过。
    常见陷阱：只跑了更小的测试，没有重新运行原始复现循环。
```

每轮 Diagnose 只需覆盖 1-2 步，根据当前状态决定下一步：

- 复现不可靠 → 回到 REPRODUCE，加强复现条件。
- 假设被排除 → 回到 HYPOTHESISE，切换到下一个假设。
- 修复无效 → 回到 HYPOTHESISE，当前假设可能错误。
- 修复引入回归 → 回到 FIX，缩小修复范围。

### 假设驱动调试

当同一失败持续存在，或某次修改没有改善信号时，列出 3-5 个排序假设。

每个假设必须包含：

- 可能原因。
- 如果为真会看到什么。
- 本轮如何验证或证伪。
- 如果为假如何排除。

每轮只测试一个主要假设。没有可证伪预测的假设只是猜测。

探针规则：

- 优先用 debugger、REPL inspection、query plan、profiler 或可重复脚本；不要先到处撒日志。
- 每条临时 debug log 必须有唯一前缀，例如 `[DEBUG-a4f2]`，交付前用前缀搜索并删除。
- 性能问题先建立 baseline 和测量点，再修改；日志通常不是性能回归的主要工具。
- 修复后必须重新运行原始复现循环，不只运行更小的回归测试。

## 递归优化 / Recursive Optimization

用于初版实现已通过真实验证、`optimization_iterations > 0`，或用户要求 N 次递归改进时。

### 原则

递归优化只能在初版实现 GREEN 后开始。目标不是抽象完美，而是在有限预算内取得可度量提升。

优化阶段要克制：尽量少抽象，多复用已有 enum、type、interface、常量和项目约定。不要为了显得高级而引入新层级、新模式或不必要的通用化。

默认参数：

```text
optimization_iterations = 12
max_no_improvement_iterations = 3
```

### 优化循环

每轮流程：

```text
1. 记录当前 baseline：验证结果、关键 diff、复杂度和已知风险
2. 选择一个最高价值优化方向
3. 做一个最小优化修改
4. 重新运行真实验证
5. 对比 baseline 和本轮结果
6. 只有质量明确提升且风险可接受时才保留
7. 没有提升或风险变大时，停止或回退本轮修改
8. 达到 N 轮或无提升阈值时结束
```

### 质量维度

每轮只优化一个主要维度：

- 实现深度：补全关键错误处理、边界保护和初版中遗漏的已知限制。
- 正确性：行为更符合需求和边界情况。
- 测试覆盖：增加真实、有价值、面向行为的测试。
- 架构一致性：更符合用户确认的架构和代码落点。
- 模块深度：interface 更小，implementation 更深。
- 可维护性：命名、职责、重复代码和依赖方向更清晰。
- 复杂度：减少分支、状态、耦合或认知负担。
- 类型复用：复用已有 enum、type、interface 或常量，减少魔法字符串和重复类型定义。
- 性能：改善真实测量结果，而不是凭感觉优化。
- 安全性：降低敏感数据、权限、校验或注入风险。
- 可观测性：让真实故障更容易定位。
- 用户体验：改善用户可观察行为。

### 每轮报告

每轮优化结果必须可比较：

```text
优化目标：
优化前：
优化后：
运行的验证：
是否保留：
保留或停止原因：
```

不要用"更干净""更好"当作证据，必须给出具体对比。

### 停止规则

在以下情况停止递归优化：

- 达到 `optimization_iterations`。
- 连续 `max_no_improvement_iterations` 轮没有明确提升。
- 优化收益低于修改风险。
- 下一步优化需要新的产品或架构决策。
- diff 超出约定范围。
- 无法重新运行真实验证。
- 优化需要引入新抽象但没有明确复用收益。

当优化暴露架构摩擦时，停止本轮自动优化，把下一步收敛为明确的接口、测试 seam 或用户决策。

## 朴素优化 / Plain Optimization

用于 GREEN 后做低风险、可验证、贴近现有代码的优化。目标是减少真实维护成本，不是显得更抽象。

### 优先做

- 复用已有 enum，替代重复的状态、类型、模式等字符串字面量。
- 复用已有 type/interface，避免同一复杂类型重复书写。
- 提取在 2 处及以上重复出现的类型别名、常量或小函数。
- 收敛命名、目录、导出和调用方式，使其符合项目既有模式。
- 删除无意义重复代码，但不改变外部行为。
- 缩小 public interface，隐藏更多 implementation 复杂度。
- 用项目已有错误类型、响应 shape、日志格式或配置模式替代局部临时写法。

### 谨慎做

- 新增抽象层。
- 新增基类、框架、插件、策略模式、工厂模式或通用注册中心。
- 为只有一个调用方的逻辑设计通用扩展点。
- 为了"优雅"把直观代码拆散到多个文件。
- 引入新依赖来节省少量代码。

禁止把"更抽象""更像架构""更高级"当作优化证据。除非它减少了真实重复、降低了维护成本，并且验证通过，否则不要保留。

### TypeScript 细则

TypeScript 项目中：

- 状态、类型、模式等离散字段优先复用已有 enum 或字符串联合。
- 复杂交叉类型在 2 处及以上使用时，优先提取为 `type` 或 `interface`。
- 不要为了类型体操牺牲调用方可读性。
- public type 是 interface 的一部分；改动前检查调用方兼容性。
- 避免在多个文件复制同一 DTO shape。

### 证据格式

每个朴素优化都给出具体对比：

```text
优化点：
优化前的重复或摩擦：
优化后的具体减少：
验证命令：
风险：
是否保留：
```

## Triage：Issue 分流工作流

用于在自动迭代启动前，对 issue 列表进行优先级排序、scope 评估和 session 分配。

### Triage 状态机

每个 issue 经过 Triage 后，进入以下状态之一：

```text
                    ┌─────────────┐
                    │  untriaged  │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ▼            ▼            ▼
        ┌─────────┐  ┌─────────┐  ┌──────────┐
        │ accepted │  │ deferred│  │ rejected │
        └────┬─────┘  └─────────┘  └──────────┘
             │
    ┌────────┼────────┐
    ▼        ▼        ▼
┌───────┐ ┌──────┐ ┌──────────┐
│ small │ │medium│ │  large   │
│ 1 轮  │ │5-10轮│ │ 10-50 轮 │
└───┬───┘ └──┬───┘ └────┬─────┘
    │        │           │
    ▼        ▼           ▼
 session  session    session
```

### Triage 维度

- **优先级**：P0（阻塞）> P1（高）> P2（中）> P3（低）> P4（待定）
- **复杂度**：small（单文件/1轮）| medium（多文件/5-10轮）| large（跨模块/10-50轮）
- **依赖关系**：blocked_by / blocks / related_to
- **验证可行性**：verifiable | partially_verifiable | not_verifiable

### Triage 流程

1. 读取所有 issue 标题和描述
2. 对每个 issue 评估优先级、复杂度、依赖、验证可行性
3. 排序并输出 Triage 结果表
4. 询问用户确认
5. 为每个 accepted 的 issue 建议 session 名
6. 用户确认后按顺序启动自动迭代

### 与自动迭代的衔接

Triage 完成后，按顺序启动自动迭代：选择第一个 accepted issue → Grill Session（或快速启动）→ 执行自动迭代 → 交付或提前停止 → 更新 Triage 状态 → 选择下一个 issue。

复杂度直接影响模式选择：small → quick/diagnose（1-5轮）；medium → medium + Autopilot（5-10轮）；large → large + Autopilot（10-50轮）。

## Zoom Out：系统视角

用于在 Context Reset Review、代码探索、PRD 验收或交付前复核时，让 Agent 从系统高度理解代码，而不是陷入局部细节。

### 何时使用

1. **Context Reset Review Gate**：`fresh_eyes_required = true` 时，先 Zoom Out 再清空细节以新人视角审视
2. **探索不熟悉代码**：先理解代码在系统中的位置、调用方和依赖
3. **实现前确认架构落点**：Grill Session 中确认代码落点符合系统结构
4. **交付前复核**：确认修改未破坏系统一致性（跨模块耦合、调用方向、重复实现、命名不一致）

### Zoom Out 输出格式

```text
## Zoom Out：系统视角

### 位置
当前代码在 [模块/服务/层] 中，属于 [系统区域]。

### 交互
- 被 [调用方列表] 调用。
- 依赖 [依赖模块列表]。
- 数据流向：[描述]。

### 影响范围
修改当前代码可能影响：
- [影响点 1]
- [影响点 2]

### 一致性检查
- 命名是否与系统其他部分一致？
- 模式是否与系统约定一致？
- 是否有重复实现？
```

### 与 Context Reset Review 的关系

Zoom Out 是 Context Reset Review 的前置步骤：Zoom Out 从系统视角俯瞰，Context Reset Review 以新人视角从零开始。两者互补：Zoom Out 防止只见树木，Context Reset Review 防止思维惯性。

Zoom Out 时必须使用领域语言（Domain Glossary 中的术语），而不是通用描述。

# Feedback Loop

用于建立快速、确定、Agent 可重复运行的 pass/fail 信号，并让每轮迭代基于真实失败推进。

## 优先级

迭代前先构造反馈闭环。

优先级：

```text
1. 最小失败测试
2. 单个 CLI 命令或脚本
3. HTTP/curl 复现
4. 可重复 fixture 输入
5. 捕获 trace / payload / 事件日志后的隔离重放
6. 临时 harness，启动系统最小子集
7. 无头浏览器脚本，断言 DOM、console 或 network
8. 非确定性问题的多次循环复现
9. 差分循环，用同一输入比较旧/新版本或不同配置
10. 人工参与循环，必须结构化记录输入、操作和输出
```

feedback loop 应满足：

- 快速：优先秒级或局部验证，不要每轮默认跑完整慢套件。
- 确定：相同输入应产生相同结果；flaky 问题要提高复现率。
- 锐利：断言具体症状，不要只检查“没有崩溃”。
- 自动化：避免人工观察，除非用户明确提供人工验证结果。

如果无法建立可信 feedback loop，停止并说明尝试过什么、还需要什么 artifact 或环境。

无法建立循环时，不要继续猜修复。向用户请求最小缺口：

- 可复现环境、命令或账号权限。
- HAR、日志 dump、core dump、trace、fixture、录屏或真实 payload。
- 允许加入临时 instrumentation 的确认。
- 可接受的沙箱替代，例如本地数据库、mock server 或只读外部服务。

## 每轮输入

每轮迭代前收集：

- 原始用户目标和成功标准。
- 当前阶段编号和名称。
- 当前循环类型：实现迭代或递归优化。
- 当前计数：`implementation_iterations`、`optimization_iterations_used`、`total_cycles`。
- 当前迭代编号和剩余预算。
- 上一轮验证命令和失败摘要。
- 上一轮优化方向以及是否保留。
- 当前 Mermaid 或纯文本进度图。
- 当前 `git diff` 摘要和允许修改范围。
- 项目已有脚本，例如 `test`、`build`、`lint`、`typecheck`。

保持上下文紧凑。优先提供失败测试名、关键 stack frame、编译错误位置、相关 diff 和最近修改文件，不要反复粘贴超长日志。

## 复现与问题对齐

修 bug 时，确认 feedback loop 复现的是用户描述的问题，而不是附近的其他失败。

确认：

- 循环呈现相同症状。
- 失败有可比对证据：错误消息、输出、状态码、指标或 UI 状态。
- 修复后重新运行原始循环，而不只是跑更小测试。

如果只能复现相似问题，停止并解释差异。

对于非确定性 bug，目标是提高复现率，而不是立即得到 100% 稳定复现。可以固定随机种子、并行循环、加压、注入延迟、缩小 timing window 或重复运行 N 次。复现率低到无法调试时，停止并请求更强 artifact。

## Diagnose 六步循环

当诊断 bug、性能回归或 flaky 测试时，使用以下六步循环：

```text
1. REPRODUCE（复现）
   目的：建立可信的 pass/fail 信号。
   输入：用户报告的 bug 描述、错误日志、复现步骤。
   输出：可重复运行的复现命令或脚本，以及明确的失败症状。
   常见陷阱：复现了相似问题但不是同一个问题，导致修复方向错误。

2. MINIMISE（最小化）
   目的：缩小问题范围，排除无关因素。
   输入：复现步骤。
   输出：最小复现用例（最少的代码、数据、步骤）。
   常见陷阱：过度简化导致问题消失，或简化后问题表现不同。

3. HYPOTHESISE（假设）
   目的：列出 3-5 个可证伪的假设，按可能性排序。
   输入：最小复现用例、代码相关区域。
   输出：排序假设列表，每个假设包含可证伪预测。
   常见陷阱：假设太模糊无法证伪，或只列一个假设导致钻牛角尖。

4. INSTRUMENT（探针）
   目的：用最小代价验证或证伪当前假设。
   输入：当前假设和可证伪预测。
   输出：探针结果（debugger 断点、REPL inspection、query plan、profiler 或临时日志）。
   常见陷阱：到处撒日志而不是用精准工具；临时日志没有唯一前缀。

5. FIX（修复）
   目的：基于验证结果做最小修复。
   输入：探针结果确认的根因。
   输出：最小代码修改。
   常见陷阱：修复了症状而不是根因；修复引入了新问题。

6. REGRESSION-TEST（回归验证）
   目的：确认修复有效且不引入回归。
   输入：原始复现步骤 + 修复后的代码。
   输出：原始复现通过 + 回归测试通过。
   常见陷阱：只跑了更小的测试，没有重新运行原始复现循环。
```

每轮 Diagnose 只需覆盖 1-2 步，根据当前状态决定下一步：

- 复现不可靠 → 回到 REPRODUCE，加强复现条件。
- 假设被排除 → 回到 HYPOTHESISE，切换到下一个假设。
- 修复无效 → 回到 HYPOTHESISE，当前假设可能错误。
- 修复引入回归 → 回到 FIX，缩小修复范围。

## 假设驱动调试

当同一失败持续存在，或某次修改没有改善信号时，列出 3-5 个排序假设。

每个假设必须包含：

- 可能原因。
- 如果为真会看到什么。
- 本轮如何验证或证伪。
- 如果为假如何排除。

每轮只测试一个主要假设。没有可证伪预测的假设只是猜测。

探针规则：

- 优先用 debugger、REPL inspection、query plan、profiler 或可重复脚本；不要先到处撒日志。
- 每条临时 debug log 必须有唯一前缀，例如 `[DEBUG-a4f2]`，交付前用前缀搜索并删除。
- 性能问题先建立 baseline 和测量点，再修改；日志通常不是性能回归的主要工具。
- 修复后必须重新运行原始复现循环，不只运行更小的回归测试。

# Feedback Loop

用于建立快速、确定、Agent 可重复运行的 pass/fail 信号，并让每轮迭代基于真实失败推进。

## 优先级

迭代前先构造反馈闭环。

优先级：

```text
1. 最小失败测试
2. 单个 CLI 命令或脚本
3. HTTP/curl 复现
4. 可重复 fixture 输入
5. 临时 harness
6. 非确定性问题的多次循环复现
```

feedback loop 应满足：

- 快速：优先秒级或局部验证，不要每轮默认跑完整慢套件。
- 确定：相同输入应产生相同结果；flaky 问题要提高复现率。
- 锐利：断言具体症状，不要只检查“没有崩溃”。
- 自动化：避免人工观察，除非用户明确提供人工验证结果。

如果无法建立可信 feedback loop，停止并说明尝试过什么、还需要什么 artifact 或环境。

## 每轮输入

每轮迭代前收集：

- 原始用户目标和成功标准。
- 当前阶段编号和名称。
- 当前循环类型：实现迭代或递归优化。
- 当前计数：`implementation_iterations`、`optimization_iterations_used`、`total_cycles`。
- 当前迭代编号和剩余预算。
- 上一轮验证命令和失败摘要。
- 上一轮优化方向以及是否保留。
- 当前 Mermaid 或纯文本进度图。
- 当前 `git diff` 摘要和允许修改范围。
- 项目已有脚本，例如 `test`、`build`、`lint`、`typecheck`。

保持上下文紧凑。优先提供失败测试名、关键 stack frame、编译错误位置、相关 diff 和最近修改文件，不要反复粘贴超长日志。

## 复现与问题对齐

修 bug 时，确认 feedback loop 复现的是用户描述的问题，而不是附近的其他失败。

确认：

- 循环呈现相同症状。
- 失败有可比对证据：错误消息、输出、状态码、指标或 UI 状态。
- 修复后重新运行原始循环，而不只是跑更小测试。

如果只能复现相似问题，停止并解释差异。

## 假设驱动调试

当同一失败持续存在，或某次修改没有改善信号时，列出 3-5 个排序假设。

每个假设必须包含：

- 可能原因。
- 如果为真会看到什么。
- 本轮如何验证或证伪。
- 如果为假如何排除。

每轮只测试一个主要假设。没有可证伪预测的假设只是猜测。

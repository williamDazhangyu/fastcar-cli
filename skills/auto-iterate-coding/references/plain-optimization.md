# 朴素优化

用于 GREEN 后做低风险、可验证、贴近现有代码的优化。目标是减少真实维护成本，不是显得更抽象。

## 优先做

- 复用已有 enum，替代重复的状态、类型、模式等字符串字面量。
- 复用已有 type/interface，避免同一复杂类型重复书写。
- 提取在 2 处及以上重复出现的类型别名、常量或小函数。
- 收敛命名、目录、导出和调用方式，使其符合项目既有模式。
- 删除无意义重复代码，但不改变外部行为。
- 缩小 public interface，隐藏更多 implementation 复杂度。
- 用项目已有错误类型、响应 shape、日志格式或配置模式替代局部临时写法。

## 谨慎做

- 新增抽象层。
- 新增基类、框架、插件、策略模式、工厂模式或通用注册中心。
- 为只有一个调用方的逻辑设计通用扩展点。
- 为了“优雅”把直观代码拆散到多个文件。
- 引入新依赖来节省少量代码。

禁止把“更抽象”“更像架构”“更高级”当作优化证据。除非它减少了真实重复、降低了维护成本，并且验证通过，否则不要保留。

## TypeScript 细则

TypeScript 项目中：

- 状态、类型、模式等离散字段优先复用已有 enum 或字符串联合。
- 复杂交叉类型在 2 处及以上使用时，优先提取为 `type` 或 `interface`。
- 不要为了类型体操牺牲调用方可读性。
- public type 是 interface 的一部分；改动前检查调用方兼容性。
- 避免在多个文件复制同一 DTO shape。

## 证据格式

每个朴素优化都给出具体对比：

```text

优化前的重复或摩擦：
优化后的具体减少：
验证命令：
风险：
是否保留：
```

# 领域语言 / Domain Language

用于在需求澄清、RCM、命名、状态记录和交付总结中使用同一套业务术语，避免 Agent 用通用词汇或实现细节替代用户能识别的概念。

## 适用场景

- Grill Session 需要确认用户说法、成功标准或验收边界。
- 从 PRD、issue、README、AGENTS.md、代码命名中提取 RCM。
- Zoom Out、Context Reset Review 或交付前需要检查术语一致性。
- 同一概念在用户、文档、代码中出现多个叫法。

## 提取规则

只收录领域术语，不收录普通工程词。

优先收录：

- 用户、业务、产品或外部系统反复使用的名词。
- 影响验收标准、状态机、权限、金额、时间、数据生命周期的概念。
- 代码中已经稳定存在，并且调用方需要理解的业务命名。
- 与其他术语容易混淆、误译或误用的词。

避免收录：

- `service`、`controller`、`helper`、`adapter` 等通用工程词，除非项目赋予了特殊含义。
- 只出现一次且不影响验收的临时变量名。
- 文件路径、类名、函数名本身；它们应放进 `relatedFiles` 或 evidence。
- Agent 为了方便解释临时发明的词。

## 术语表格式

每个术语至少记录：

| 字段 | 含义 |
| --- | --- |
| `term` | 用户或项目中使用的主要中文/本地术语 |
| `english` | 需要时使用的英文或代码侧对应词 |
| `definition` | 可验收、不可循环定义的一句话 |
| `source` | 来源文档、用户原话、代码位置或确认记录 |
| `notes` | 边界、反例、同义词、禁用叫法 |

示例：

| term | english | definition | source | notes |
| --- | --- | --- | --- | --- |
| 试算 | quote preview | 在不创建订单的情况下计算价格、折扣和费用 | docs/pricing.md | 不等同于付款 |
| 生效窗口 | effective window | 配置允许影响线上行为的开始和结束时间 | issue-42 | 不等同于展示时间 |

## 使用规则

- RCM 的 `summary` 使用领域术语描述用户可见行为；实现文件和函数名放入证据字段。
- 代码命名优先沿用已有领域词，不随意改成同义词。
- 交付总结、验证结果和用户提问使用同一术语，必要时同时给出代码侧名称。
- 发现同义词、歧义词或术语与实现不一致时，先记录为 `needs_confirmation`，不要静默选择一个含义。
- 术语定义影响架构、数据模型、权限或兼容性时，升级为 decision；只有 hard-to-reverse、出乎意料或存在真实取舍的决定才需要 ADR。

## 更新时机

- 启动握手：从用户原话和项目文档提取候选术语。
- 代码探索：发现重复业务命名或边界概念时补充来源。
- 实现中：发现术语影响测试 seam、状态流转或验收路径时更新。
- 交付前：检查 RCM、DoD、文档、命名和交付总结是否使用同一术语。

## 反模式

- 用文件名或层名替代用户可见需求，例如把"试算结果应包含服务费"写成"修改 PricingService"。
- 把术语表写成完整业务百科，导致 Agent 每轮都加载低价值背景。
- 未确认歧义词就继续实现，例如"账户"可能指登录账号、钱包账户或结算账户。
- 把一次性实现细节沉淀成领域术语。

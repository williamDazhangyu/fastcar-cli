# 停止条件

用于判断是否继续实现迭代、继续递归优化，或提前停止并请求用户决策/资源。

## 停止边界

停止边界必须按用户提供的完整目标、流程清单、Runbook、MVP 清单、成功标准和显式非目标判断。单个 Phase、子任务、最小纵切、当前失败信号清零或局部验证通过，不等于整体完成。

当任务包含多阶段清单时，必须持续维护 `完整任务清单 / 已完成任务 / 当前任务 / 剩余任务 / 整体完成状态`。只要 `剩余任务` 非空且没有触发必须停止条件，就必须继续迭代到下一项，不能按成功交付停止。

## 必须停止

以下情况必须停止：

- 完整任务清单、Runbook 或用户目标中的所有关键成功标准满足，且对应验证通过。
- 达到 `max_iterations`。
- 达到 `optimization_iterations`。
- 连续 `max_no_progress_iterations` 轮实现迭代没有失败改善。
- 连续 `max_no_improvement_iterations` 轮优化没有质量提升。
- 当前 diff 超过大小或范围限制。
- 产品行为、API 兼容性、迁移、权限或架构需要用户决策。
- 需求不明确，或多个合理实现会影响兼容性。
- 必要时，架构理解、module 落点或脚手架用法未确认。
- 真实验证需要数据库、密钥、外部服务或测试环境，但用户没有确认沙箱替代。
- 验证结果退化：通过测试减少、构建错误增加、出现无关失败。
- 没有可靠 feedback loop，或它不能对齐用户描述的问题。
- 优化收益低于风险。
- 看门狗触发 `stop`，例如连续无进展、状态漂移无法消除、已有验证未运行、交付成果不可验证或交付可验证性为 `unknown`。
- 临时 debug instrumentation、一次性 harness、原型路由或未吸收原型仍存在，且没有用户确认的保留理由。
- 架构摩擦已确认，继续自动 patch 会扩大修改范围或破坏 public interface。

以下情况不能作为成功停止理由：

- 当前 Phase 通过，但后续 Phase、Runbook 条目或 MVP 项仍未完成。
- 一个最小纵切通过，但用户要求的是完整模块、完整项目或“直到完成所有任务”。
- 测试、构建或 e2e 通过，但 Definition of Done 仍有 `not_verified` 或 `blocked`。
- 状态文件中的 `下一步`、`剩余任务`、`未完成 TODO` 或 `已知限制` 仍包含用户目标内的必要工作。
- 最终交付成果无法被当前环境验证，或缺少必须的验收数据、命令、权限、外部服务、数据库、密钥、沙箱替代或用户决策。
- 原型能运行或某个方案看起来可行，但尚未被用户确认、吸收为生产实现并完成验证。

## 停止时输出

提前停止时必须输出：

```text
已完成内容：
停止原因：
当前停止阶段：
实现迭代总数：
递归优化轮次：
总循环次数：
最后一个关键错误：
已运行的验证：
未运行验证及原因：
交付可验证性：verifiable / partially_verifiable / not_verifiable / unknown
看门狗触发原因：
未清理临时产物：
架构摩擦：
需要用户提供的资源或决策：
建议的下一步：
```

不要把停止包装成完成。完成与提前停止必须明确区分。

## 继续前检查

继续下一轮前问自己：

- 这一轮是否有新的失败信号？
- 修改范围是否仍和用户目标直接相关？
- 是否仍有可重复 feedback loop？
- 是否正在通过 mock、硬编码或跳过测试掩盖失败？
- 下一步是否需要用户决策？
- 交付成果是否真的能被当前环境验证？
- 看门狗状态是否为 clear？
- 是否还留有 debug 日志、harness、原型或临时路由？
- 是否已经出现需要用户确认的架构摩擦？

如果任一答案显示风险上升，停止并汇报。
